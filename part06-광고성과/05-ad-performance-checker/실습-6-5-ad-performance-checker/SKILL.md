# Part 6 / 6-5 · 광고 성과 임계값 감시 실습 : 소개부터 침묵·경고까지

> 시간당 자동으로 3매체 지표를 체크해 **임계값 위반 시에만 알리는** 감시 에이전트를 가동한다.
> 핵심 학습 목표 : ① 침묵 원칙 (위반 0건 = 무소식) ② 임계값의 .env 외부화 ③ 권고만, 자동 중지는 없음.
> ⛔ 자동 실행 금지. 단계마다 사용자 답을 받은 뒤에만 다음으로.

## 진행 흐름 (3 단계)

```
1단계  소개   하는 일 + 연결 소스                          ⏸ 답 대기
2단계  실행   사전 점검 → 작동 구조 → 데이터 준비 → 실제 실행  ⏸ 답 대기
3단계  활용   내 업무 적용법 + 산출물 + 다음 클립            ⏸ 선택
```

---

## 1단계 · 소개

```
🚨 광고 성과 임계값 감시 에이전트를 소개합니다.

본 흐름의 핵심 : "매시간 들여다보는 건 사람이 할 일이 아니다. 문제가 있을 때만 울리는 알람을 깐다"

🧩 하는 일
  · 매시간 (cron) 또는 수동 1회로 3매체 최신 지표를 자동 조회
  · 임계값 7종과 비교해 위반만 골라냄 (WARN / CRITICAL 이중 기준)
  · 위반 0건 → 침묵 / 위반 있음 → #marketing-alerts 에 경고 + 권고 액션
  · 동일 위반 3시간 연속 → @멘션 에스컬레이션

🔌 연결 소스
  ad-performance-checker
   ├─ .env                       ← 임계값 7종 (판정 기준 · 코드 밖 외부화)
   ├─ Meta · Google Ads MCP      ← 실계정 모드 최신 지표 (샘플 모드는 불필요)
   └─ Discord #marketing-alerts  ← 경고 전용 채널 (일반 알림과 등급 분리)

💡 핵심 포인트
  · 침묵 원칙 : 위반이 없으면 아무것도 안 보낸다 (알림 피로 = 알림 무시의 시작)
  · 임계값은 코드가 아니라 .env : 기준이 바뀌어도 에이전트는 안 건드린다
  · 자동 중지는 없다 : "일시정지 권고"까지만 · 예산 변경은 사람 승인 영역

👉 2단계로 갈까요? (실제 실행을 직접 봅니다)
```

📐 **스킬? 에이전트?** → 도구(Meta·Google Ads) + 트리거(매시간 cron) + 산출(Discord 경고)이 있으니 🤖 **에이전트**. 만드는 법은 3-1의 **에이전트 5요소** 골격 그대로.

⏸ 답 받고 2단계로.

## 2단계 · 실행

### 2-0 · 사전 점검 (+ 임계값 .env 세팅)

```bash
grep -q "THRESHOLD_ROAS_WARN" .env 2>/dev/null && echo "임계값 ✅" || echo "임계값 미설정"
test -f "discord-bot/webhook-config.json" && echo "webhook ✅"
```

```
🔧 6-5 의존 항목 점검 결과

  | # | 항목 | 용도 | 상태 | 해결 |
  |---|---|---|---|---|
  | ① | 임계값 7종 (.env)         | 판정 기준        | {✅ / ❌} | 1분 (자동 등록) |
  | ② | Meta·Google Ads MCP      | 실계정 모드 조회 | {✅ / ❌} | 6-1·6-2 절차 |
  | ③ | Discord #marketing-alerts | 경고 채널        | {✅ / ❌} | 채널 생성 + webhook |

  ❌ ① 은 sample-data/thresholds.env.sample 의 7종을 .env 에 자동 추가 후 재확인.
     (ROAS_WARN 2.0 / BLOCK 1.0 · CPA_WARN 30000 / BLOCK 50000 · CTR_WARN 0.01 / BLOCK 0.005 · BUDGET_WARN 1.1)
  ③ 경고는 일반 알림과 다른 전용 채널 권장 (알림 등급 분리).
  샘플 모드는 ② 없이 진행 가능.
```

⏸ 세팅 확인 받고 2-1로.

### 2-1 · 작동 구조

```
🔄 에이전트 워크플로
  [매시간 (cron) 또는 수동 1회]
    → 3매체 최신 지표 조회
    → 임계값 비교 :
       ROAS < 2.0 WARN · < 1.0 CRITICAL / CPA > 3만 WARN · > 5만 CRITICAL
       CTR < 1% WARN / 일예산 소진 > 110% WARN · > 130% CRITICAL
    → 위반 0건 : 침묵 (아무것도 안 보냄)
    → 위반 있음 : #marketing-alerts 에 캠페인·지표·현재값·임계값 + 권고 액션
    → 동일 위반 3시간 연속 : @멘션 에스컬레이션

📄 에이전트 정의에서 새로 볼 부분 : open "agents/part6-ads/ad-performance-checker.md"
  ① 임계값이 코드가 아니라 .env : 기준 변경에 에이전트 수정 불필요 (설정 외부화 재등장)
  ② 침묵 원칙이 명문화 : "위반 없으면 발송하지 않는다"
  ③ 에스컬레이션 단계 : 1회 위반 (경고) → 3시간 연속 (@멘션) · 등급 상승 설계

퀴즈 : "왜 ROAS 1.5 캠페인에 CRITICAL 이 아니라 WARN 일까?" (답: WARN 2.0 / CRITICAL 1.0 이중 기준)
```

⏸ 답 받고 2-2로.

### 2-2 · 데이터 준비 (임계값 + 성과 픽스처)

```
1. .env 임계값 7종 표로 확인 (WARN / CRITICAL 이중 구조 설명)
2. 성과 데이터 :
   - [A] 샘플 : sample-data/meta-ads-performance.json (캠페인 3 · 광고세트 6)
            → 리치 캠페인 ROAS 0.43 이 위반 후보 (목적이 '도달'이라 맥락 주석 동반)
   - [B] 실계정 : 6-1·6-2 연결 계정 최신 지표 (읽기 전용)
3. 시뮬레이션 예고 : "정상 기준 1회 → 임계값 10.0 으로 올려 위반 강제 1회, 두 장면 비교"
```

⏸ OK 받고 2-3으로.

### 2-3 · 실제 실행 (침묵 → 위반 강제 → 경고)

```
▶ 3회 실행으로 침묵과 경고를 대비:

  1회차 · 정상 임계값 (ROAS WARN 2.0):
    → 체크 결과 표 : 리치 캠페인 ROAS 0.43 → CRITICAL 1건 검출
      (목적이 '도달' 캠페인이므로 권고문에 맥락 주석이 붙는 것 확인)
    → #marketing-alerts 에 경고 1건 발송

  2회차 · 침묵 시연 (임계값을 일시적으로 ROAS WARN 0.1 로 낮춤):
    → 위반 0건 → "발송 없음" : 터미널엔 체크 로그만, 디스코드는 조용함 (시연 하이라이트)

  3회차 · 시뮬레이션 (THRESHOLD_ROAS_WARN=10.0 으로 위반 강제):
    → 전 캠페인 위반 → 다건 경고 포맷 + 에스컬레이션 조건 설명 (알림 장면 결정적 재현)
    → 시연 후 임계값 원복 ⏸ 원복 확인 필수
```

⏸ 각 회차 결과 확인 후 3단계로.

