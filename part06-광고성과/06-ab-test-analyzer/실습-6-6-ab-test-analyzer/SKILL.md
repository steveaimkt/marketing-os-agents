# Part 6 / 6-6 · A/B 테스트 분석 에이전트 실습 : 소개 → 실행 → 활용

> A/B 테스트 결과를 **Bayesian(베이지안) 승리 확률로 판정하고 승리 패턴을 자산화**하는 에이전트를 가동한다.
> 핵심 학습 목표 : ① 승리 확률 판정 (클릭 차이 ≠ 승리) ② 표본 부족 인지 ③ 패턴 라이브러리 자산화.
> ⛔ 자동 실행 금지. 단계마다 사용자 답을 받은 뒤에만 다음으로.

```
1단계  소개   에이전트가 하는 일 + 연결 소스         ⏸ 답 대기
2단계  실행   작동 구조 + 설계 확인 + 데이터 + 실행    ⏸ 답 대기
3단계  활용   내 업무 적용법 + Part 6 마무리         ⏸ 선택
```

## 1단계 · 소개

```
🧪 A/B 테스트 분석 에이전트 실습을 시작합니다.

이 에이전트가 하는 일 :
  "B 가 좀 더 잘 나왔다 ≠ B 의 승리. 확률로 판정하고, 승리는 자산으로 쌓는다"

⏱ Before vs After
  · Before : 전환 수 눈대중 비교 → 우연을 승리로 오판 → 잘못된 학습 누적
  · After  : 승리 확률 % + 표본 충분성 판정 + 승리 패턴 라이브러리 자동 적재

핵심 포인트 3가지 :
  ① 승리 확률 판정  — 클릭·전환이 더 높다고 이긴 게 아니다 (확률로 말한다)
  ② 표본 부족 인지  — 퍼센트가 커 보여도 표본이 작으면 판정을 보류한다
  ③ 패턴 자산화     — 이긴 이유를 재사용 가능한 형태로 라이브러리에 남긴다
```

```
🔌 연결 소스 (이 실습에서 실제로 의존하는 것)

  ├─ Notion 통합          승리 패턴 라이브러리 적재        ← 필수
  ├─ Discord webhook      판정 3건 발송                   ← 필수
  └─ Meta·Google Ads MCP  실계정 활성 테스트 조회          ← 실계정 모드만

  ⚠ 샘플 모드는 Ads MCP 없이 진행 가능 (테스트 3건 픽스처 내장).
```

  ③ 판정문에 "다음 행동" 이 항상 동반 : 채택 / 다른 차원 / 추가 수집
  ④ 승리 패턴의 일반화 적재 : "긴급형 > 혜택형 +37%" 식 재사용 가능한 형태

### 2-2 · 테스트 설계 확인 (A·B를 어떻게 정하나)

```
🅰️🅱️ 판정은 설계가 옳아야 의미가 있다. 6숫자를 넣기 전 이 4개를 확인 :

  ① A = 챔피언(지금 쓰는 것) · B = 챌린저(새로 시험하는 것)
     → A 는 기준선. B 가 A 를 이겨야 교체.
  ② 딱 한 변수만 다르게 (통제)
     → 헤드라인만, 또는 형식만. 두 개 바꾸면 "무엇이 원인"인지 못 가림.
  ③ 나머지는 전부 동일 — 예산·타겟·기간·게재위치
     → 이게 어긋나면 A/B 가 아니라 교차 분석이 됨.
  ④ 무슨 차원을 바꿀지 우선순위 : 소구·헤드라인 > 크리에이티브 형식 > 타겟 > LP
     → 보통 소구가 임팩트 최대. 임팩트 큰 차원부터.

  샘플 3건이 딱 이 규칙대로 설계됨 :
     AB-001 헤드라인만 · AB-002 형식만 · AB-003 LP만 (각각 한 변수)

  ⚠ 실계정 셋업 : Meta 는 실험 등록(ads_experiment_abtest_create_test)으로
     예산·타겟을 자동 균등 분배. 수동이면 두 광고를 같은 조건으로 맞춰야 함.
```
