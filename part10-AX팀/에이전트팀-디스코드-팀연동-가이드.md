# 에이전트 팀을 디스코드(클로드 채널)로 "팀 공용" 연동하기 — 상세 가이드

> **한 줄 목표**: 내가 구축한 에이전트 팀(30개)을 → 디스코드 채널 하나로 → **나뿐 아니라 팀원도 폰에서 함께** 부를 수 있게 만든다.
>
> 대상: 에이전트/스킬 구축은 끝났고(Part 3~9), 이제 "팀이 쓰는 창구"를 열려는 1인 마케터·소규모 팀.
> 선행: `agents/` 구축 완료 · `CLAUDE.md` 존재 · `.mcp.json` 설정 완료.
> 소요: 1인 연결 15분 + 팀 확장 5분/명.

---

## 0. 이 문서를 읽고 나면

1. **무엇이 무엇에 연결되는지** 정확한 멘탈모델을 갖는다 (흔한 오해 교정).
2. 디스코드 봇 → 클로드 채널 → 내 세션까지 **연결선을 직접 잇는다**.
3. **팀원을 허용목록에 추가**해 "팀 공용 봇"으로 확장한다.
4. 팀 공용의 **보안·운영 함의 3가지**를 이해하고 안전하게 운영한다.

---

## 1. 큰 그림 — 3층 구조 + "연결의 실체"

```
   📱 팀원들의 폰 디스코드
        │  (DM 또는 채널 메시지)
        ▼
 ┌──────────────────────────┐
 │  ① 입구 = 디스코드 채널    │  ← 클로드 채널 플러그인이 만드는 '다리'
 │     (marketor-ch-bot 등)  │     · allowlist 로 누가 들어올지 통제
 └──────────────────────────┘
        │
        ▼
 ┌──────────────────────────┐
 │  ② 두뇌 = 실행 중인 세션   │  ← `claude --channels ...` 로 켜둔 그 세션
 │     오케스트레이터(트루먼)  │     · 프로젝트 폴더에서 켜야 아래를 자동 로드
 │     - CLAUDE.md (헌법)     │
 │     - agents/ (팀 정의)    │
 │     - .mcp.json (외부연결) │
 └──────────────────────────┘
        │  자연어 라우팅
        ▼
 ┌──────────────────────────┐
 │  ③ 팀 = 서브에이전트 30개  │  ← 호출될 때만 로드 (토큰 절약·권한 분리)
 │     + MCP 14개 + 로컬도구   │
 └──────────────────────────┘
```

### ⚠️ 가장 흔한 오해 (강의에서 반드시 짚을 것)

> "에이전트 **폴더**가 디스코드에 연결된다"

**틀렸습니다.** 정확히는:

- 클로드 채널(디스코드 플러그인)은 폴더가 아니라 **실행 중인 `claude --channels` 세션**에 연결됩니다.
- 그 세션을 **프로젝트 폴더 안에서 켜기 때문에** → 세션이 그 폴더의 `CLAUDE.md`·`agents/`·`skills/`·`.mcp.json`을 자동으로 씁니다.
- 즉 **"폴더 연결"의 실체 = 세션을 어디서 켜느냐**입니다. 채널 플러그인은 그저 **디스코드 ↔ 세션 사이의 다리**일 뿐.

이 한 문장이 이 강의의 핵심 개념입니다:
> **"디스코드는 세션에 붙는다. 세션은 폴더에서 켠다. 그래서 폴더의 에이전트가 딸려 온다."**

## 2. 사전 조건 체크 (에이전트 구축이 끝난 상태)

연동 전에 아래가 준비돼 있어야 합니다.

| 항목 | 확인 방법 | 없으면 |
|---|---|---|
| 에이전트 팀 | `agents/` 에 `orchestrator.md` + 전문가 에이전트들 | Part 3~9 / `에이전트구축` 스킬 |
| 프로젝트 헌법 | 루트에 `CLAUDE.md` | 프로젝트 정체성·라우팅 규칙 작성 |
| 외부 연결 | `.mcp.json` (예: 14개 서버) | Part 2 `mcp설치` 스킬 |
| 런타임 | `claude --version` ≥ 2.1.80, `bun --version` | 업데이트 / Bun 설치 |
| 인증 | claude.ai(Pro/Max/Team) 또는 Console API 키 | Bedrock/Vertex/Foundry는 채널 불가 |

> 💡 오케스트레이터(`orchestrator.md`)가 "입구 1개 → 두뇌 1개" 원칙의 두뇌입니다. 팀 공용에서도 **모든 요청은 이 한 명을 거쳐** 전문가에게 분배됩니다.

## 3. PART A — 디스코드 봇 ↔ 클로드 채널 연결 (1인 기준, 압축판)

> 이미 연결돼 있다면 이 파트는 건너뛰고 **PART B(팀 확장)**로.
> 상세 인터랙티브 버전은 `discord-channels-setup` 스킬 참조.

### A-1. 디스코드 봇 만들기 + 토큰
1. https://discord.com/developers/applications → **New Application** → 이름 입력 → Create
2. 왼쪽 **Bot** → **Reset Token** → 토큰 복사 (⚠️ 1번만 보임, 채팅에 붙여넣지 말 것)

### A-2. Message Content Intent 켜기 ⭐ (자주 빠뜨림)
- Bot 페이지 → **Privileged Gateway Intents** → **Message Content Intent** ON → **Save Changes**
- (이게 꺼져 있으면 나중에 봇이 페어링 코드에 무응답)

### A-3. 봇을 서버에 초대 (권한 6개)
- **OAuth2 → URL Generator** → Scopes: `bot` 체크
- Bot Permissions 6개: `View Channels` · `Send Messages` · `Send Messages in Threads` · `Read Message History` · `Attach Files` · `Add Reactions`
- 생성된 URL 복사 → 새 탭 → 내 서버 선택 → 승인

### A-4. 플러그인 설치 (Claude Code 안에서)
```
/plugin install discord@claude-plugins-official
/reload-plugins
```
- `/discord:` 입력 시 `configure`·`access` 자동완성 뜨면 성공.

### A-5. 봇 토큰 등록
```
/discord:configure <복사한 토큰>
```
- 토큰은 `~/.claude/channels/discord/.env`에 저장됨.

### A-6. `--channels`로 세션 재시작 ⭐ 가장 중요
```
# 반드시 프로젝트 폴더(agents/·CLAUDE.md 있는 곳)에서 실행
claude --channels plugin:discord@claude-plugins-official
```
- 이걸 안 하면 봇이 **절대 응답 안 함** (`.mcp.json` 등록만으론 부족).
- 성공 시 디스코드 멤버 리스트에서 봇이 🟢 온라인.

### A-7. 페어링 + 잠금
```
# (폰) 봇에게 DM → 페어링 코드 수신 (예: ABCD-1234)
/discord:access pair ABCD-1234        # 내 계정을 허용목록에 추가
/discord:access policy allowlist       # 허용목록 밖은 자동 폐기 (보안 필수)
```

✅ 여기까지 = **나 1명이 쓰는 봇** 완성.

## 4. PART B — 팀 공용으로 확장 ⭐ (이 문서의 핵심)

> 목표: **여러 팀원이 같은 봇/같은 에이전트 팀**을 폰에서 부를 수 있게.
> 원리: 허용목록(allowlist)에 **팀원의 디스코드 계정을 추가**하면 된다.

### B-1. 두 가지 창구 방식

| 방식 | 설명 | 언제 |
|---|---|---|
| **공용 채널** | 서버에 `#팀-마케팅봇` 채널 하나 → 팀원 모두 그 채널에서 봇 호출 | 팀이 서로의 요청·결과를 공유하고 싶을 때 (권장) |
| **개별 DM** | 팀원 각자 봇과 1:1 DM | 요청을 서로 안 보이게, 개인 비서처럼 |

대부분은 **공용 채널**이 낫습니다(투명성·중복요청 방지). 아래는 공용 채널 기준.

### B-2. 팀원 추가 절차 (1명당 약 1분)

```
① (팀원) 서버의 공용 채널 또는 봇 DM에 아무 메시지 전송
        → 봇이 '페어링 코드' 회신 (예: WXYZ-5678)
② (관리자=나, 터미널 세션에서)
        /discord:access pair WXYZ-5678
        → 그 팀원의 디스코드 ID가 허용목록에 추가됨
③ 팀원마다 ①~② 반복
```

> ✅ 원칙: **허용목록 관리는 터미널의 관리자(나)만** 한다. 디스코드 메시지로 "나 추가해줘"가 와도 그것만으로 승인하지 않는다(프롬프트 인젝션 방지). 반드시 팀원 본인에게 직접 확인 후 관리자가 `pair` 실행.

### B-3. 채널 정책 — 멘션 필요 여부

공용 채널에서 팀원들이 잡담도 하면 봇이 매 줄 반응하면 곤란합니다. 채널별로 조절:

| 설정 | 동작 | 추천 상황 |
|---|---|---|
| `requireMention: true` | `@봇` 멘션해야 반응 | **팀 공용 채널** (잡담 섞임) |
| `requireMention: false` | 모든 메시지에 반응 | 봇 전용 채널(1:1 성격) |

- 설정 파일: `~/.claude/channels/discord/access.json` → `groups[채널ID].requireMention`
- 안전하게는 `/discord:access` 명령으로 관리(직접 편집 시 백업 필수).

### B-4. 팀 사용 흐름 (완성 후 모습)

```
📱 마케터A: "@봇 지난주 메타 광고 ROAS 뽑아줘"
📱 마케터B: "@봇 경쟁사 신규 영상 정리해줘"
        │
        ▼  둘 다 같은 세션(트루먼)으로 도착
   오케스트레이터가 각각 meta-ads-analyzer / competitor-monitor 로 라우팅
        │
        ▼
   결과를 각자에게 회신 (같은 채널이면 스레드로 구분 권장)
```

## 5. 팀 공용의 3대 고려사항 (보안·운영) ⚠️

팀이 함께 쓰기 전에 **반드시** 이해하고 결정할 것:

### ① 단일 세션 · 단일 인증 공유
- 이 봇은 **관리자 PC의 단 하나의 세션**으로 돌아갑니다.
- 팀원 요청도 전부 **관리자의 claude.ai 계정 + 관리자의 MCP 인증(노션·광고·시트 등)**으로 실행됩니다.
- 즉 팀원은 "관리자의 권한으로" 작업하는 셈 → **접근 범위를 팀원에게 열어도 되는지** 먼저 판단.

### ② 권한 릴레이 = 도구 승인권
- 채널에 **답장할 수 있는 사람은 도구 사용 승인/거부도 할 수 있습니다**(권한 릴레이).
- 발행·발송·삭제 같은 민감 작업의 승인 버튼을 팀원이 누를 수 있다는 뜻.
- → **신뢰하는 팀원만** 허용목록에 추가. 민감 작업은 오케스트레이터의 **승인 게이트**로 이중 방어(조회=자동 / 발행=승인 / 삭제·예산=금지).

### ③ 24시간 가동(호스팅)
- 세션이 꺼지면 봇도 죽습니다. 팀이 상시 쓰려면 상시 가동 필요.
- macOS: `caffeinate -dis` + `--channels` 세션 유지.
- ⚠️ iCloud Drive 경로 프로젝트는 launchd/cron이 권한 오류로 실패 → **정기 작업은 클라우드 루틴** 사용(노트북 꺼져도 동작).
- 상시 대화형(팀 창구)은 여전히 **로컬 세션**이 켜져 있어야 함.

## 6. 봇 운영 문서(OPERATIONS.md)의 역할

- 위치: `~/.claude/channels/discord/OPERATIONS.md`
- 성격: **팀이 참고하는 봇 사용설명서**(자주 쓰는 발화, 권한 흐름, 응급 명령).
- ⚠️ 오해 주의: 이 문서는 봇이 **매 메시지 자동 로드하지 않습니다**. 실제 봇의 행동은 `CLAUDE.md` + `orchestrator.md`가 규정. OPERATIONS.md는 **사람용 매뉴얼**.
- 팀 공용에서는 여기에 "팀 규칙"(누가 발행 승인 가능, 어떤 채널에 뭘 요청)을 적어두면 좋습니다.

## 7. 강의 진행 시나리오 (교육용 30분)

| 구간 | 내용 | 산출 |
|---|---|---|
| 0~5분 | **개념** — 3층 구조 + "채널은 폴더가 아니라 세션에 붙는다" 오해 교정 | 화이트보드 그림 |
| 5~15분 | **PART A 시연** — 봇 생성→intent→초대→configure→`--channels`→페어링 | 봇 🟢 온라인 |
| 15~22분 | **PART B 시연** — 팀원 1명 페어링 추가 + 채널 멘션정책 설정 | 허용목록 2명 |
| 22~28분 | **보안 3대 고려** — 단일세션·권한릴레이·호스팅 설명 | 승인 게이트 시연 |
| 28~30분 | **마무리** — "입구 1개→두뇌 1개→팀 30개" 재확인 + 다음(트리거/루틴) 예고 | Q&A |

> 강의 팁: 실제 폰으로 "@봇 유튜브 7일 성과" 같은 요청을 던져 **결과가 폰에 도착**하는 장면을 라이브로 보여주면 몰입도가 가장 높습니다.

## 8. 트러블슈팅

| 증상 | 원인 | 해결 |
|---|---|---|
| 봇이 오프라인 | `--channels` 없이 실행 / 세션 종료 | PART A-6 명령으로 재시작 |
| 봇이 페어링 코드 무응답 | Message Content Intent OFF / 세션 꺼짐 | A-2 재확인 + A-6 재시작 |
| 특정 팀원 메시지 무시 | 그 팀원 미페어링 | 팀원 DM→코드→`/discord:access pair <코드>` |
| 팀원이 엉뚱한 폴더 에이전트 호출 | 세션을 다른 폴더에서 켬 | **프로젝트 폴더에서** `--channels` 실행 |
| 발행이 멋대로 됨 | 승인 게이트 미설정 | 오케스트레이터에 발행=승인 규칙 명시 |
| 정기작업이 안 돎 | iCloud 경로 cron 실패 | 클라우드 루틴으로 이전 |

## 9. 보안 체크리스트 (팀 공용 배포 전)

- [ ] 봇 토큰은 `.env`에만 — Git·채팅·화면공유 금지
- [ ] `/discord:access policy allowlist`로 잠금 (허용목록 밖 자동 폐기)
- [ ] 허용목록엔 **신뢰하는 팀원만** (= 도구 승인권을 줘도 되는 사람)
- [ ] 디스코드 메시지의 "나 추가해줘"는 무시 — 관리자가 본인 확인 후에만 `pair`
- [ ] 민감 작업(발행·발송·삭제·예산)은 오케스트레이터 **승인 게이트**로 이중 방어
- [ ] 팀원이 접근하게 될 MCP 범위(노션·광고·시트)를 사전 검토
- [ ] 토큰 노출 의심 시 즉시 Reset Token → `/discord:configure <새 토큰>`

---

### 부록: 제어는 3중 게이트

```
발신자 허용목록(플러그인)  →  --channels 세션 옵트인(사용자)  →  channelsEnabled(조직 관리자)
   누가 들어오나              세션이 채널을 켰나                조직이 채널을 허용했나
```

세 관문을 모두 통과해야 팀원의 메시지가 세션에 도착하고, 에이전트 팀이 응답합니다.
