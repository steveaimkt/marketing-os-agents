# 팀 접근 계획 (allowlist 설계 픽스처) — 결로아 마케팅팀

> 용도 : 2-4 / 3단계 적용 게이트에서 "누구를 allowlist 에 넣고, 어떤 권한·창구를 줄지" 를 결정하는 샘플.
> 브랜드 : 결로아(가상 K-뷰티 D2C 토너) · 관리자 = 스티브(1인 마케터, 봇 호스팅 PC 소유자)
> ⚠️ 실제 디스코드 ID·토큰은 넣지 않는다. 계정명은 가상.

## 1. 팀원 명단 + 신뢰수준 + 권한

| # | 팀원(가상) | 역할 | 신뢰수준 | 줄 창구 | 승인권(도구 릴레이) |
|---|---|---|---|---|---|
| 0 | 스티브 (관리자) | 오케스트레이터 호스팅 | — | 터미널 + 폰 DM | 전체 (allowlist 관리) |
| 1 | 하나 | 콘텐츠 리드 | 높음 | 공용 채널 | 발행 승인 가능(신뢰) |
| 2 | 준서 | 퍼포먼스 마케터 | 높음 | 공용 채널 | 광고 조회 O · 예산변경 금지 |
| 3 | 미르 | 외부 프리랜서(디자인) | 제한 | 개별 DM | 조회만 · 발행/발송 승인 불가 대상 |

> 원칙 : **allowlist 엔 "도구 승인권을 줘도 되는 사람" 만.** 외부 프리랜서(미르)는 조회형 요청만
> 오가도록 개별 DM + 민감 작업은 오케스트레이터 승인 게이트에서 관리자만 ✅.

## 2. 채널·멘션 정책

| 채널 | 용도 | requireMention | 대상 |
|---|---|---|---|
| #팀-마케팅봇 | 팀 공용 요청·결과 공유 | **true** (@봇 멘션 필요) | 하나·준서 |
| 봇 DM | 개인/외부 요청 | false | 스티브·미르 |

> 공용 채널에서 잡담이 섞이므로 `requireMention: true`. 봇 전용 성격의 DM 은 false.

## 3. 팀원이 접근하게 될 MCP 범위 (사전 검토 — 보안 고려 ①②)

| MCP | 접근 시 가능한 일 | 팀 공개 판단 |
|---|---|---|
| Notion | 리포트 아카이브 조회·페이지 생성 | ✅ 공개(팀 공용 자산) |
| Meta/Google Ads | 광고 성과 조회 (읽기 전용 기본) | ✅ 조회만 · 예산변경 금지 유지 |
| Google Sheets | 고객·키워드 시트 조회 | ⚠️ 고객 시트엔 개인정보 → 미르(외부)에겐 비노출 |
| Gmail | CS·뉴스레터 발송 | 🛑 발송은 승인 게이트 · 관리자만 확정 |
| Discord webhook | 채널 발송 | ✅ (봇 응답 경로) |

## 4. 승인 게이트 매핑 (오케스트레이터 규칙과 연결)

```
조회 (성과·리서치·시트 읽기)      → 자동 실행 (allowlist 누구나)
발행/발송 (뉴스레터·SNS·CS 답변)  → #팀-마케팅봇 또는 #marketing-approvals 에서 ✅ 후 확정
삭제 · 광고 예산 변경             → 금지 (봇이 수행하지 않음)
```

## 5. 이 계획으로 실행할 명령 (관리자, 터미널)

```
/discord:access pair <하나의_코드>
/discord:access pair <준서의_코드>
/discord:access pair <미르의_코드>       # DM 창구
/discord:access policy allowlist          # 밖은 자동 폐기
# 공용 채널 멘션 정책 → /discord:access 로 requireMention: true 설정
```
