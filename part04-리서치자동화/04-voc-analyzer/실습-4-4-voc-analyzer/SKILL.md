# Part 4 / 4-4 · VoC 분석기 실습 : 소개 → 실행 → 활용

> 리뷰·SNS 멘션·CS 메일을 모아 **5축으로 분류하고 페인포인트를 클러스터링**하는 에이전트를 가동한다.
> 핵심 학습 목표 : ① 5축 분류 (감정·카테고리·긴급도·신규성·임팩트) ② 클러스터링 ③ 해결책 제안.
> ⛔ 자동 실행 금지. 단계마다 사용자 답을 받은 뒤에만 다음으로.

## 진행 흐름 (3 단계)

```
1) 소개   하는 일 + Before/After + 연결 소스 + 핵심 포인트     ⏸ 답 대기
2) 실행   사전 점검 → 작동 구조 → 데이터 준비 → 실제 실행      ⏸ 답 대기
3) 활용   내 업무 적용법 + 산출물 + 다음 클립                   ⏸ 선택
```

---

## 1단계 · 소개

```
🗣 VoC 분석기 실습을 시작합니다.

본 흐름의 핵심 : "흩어진 고객의 소리 500건을 5축 좌표로 찍으면 우선순위가 보인다"

🔧 하는 일
  자사몰 리뷰 + 올리브영 + 인스타 멘션 + X 검색 + Gmail CS 를 한데 모아
  5축으로 분류하고, 반복되는 불만을 묶어 TOP 페인포인트 + 해결책으로 정리합니다.

⏱ Before vs After
  · Before : 리뷰 훑어보기 + CS 메일 따로 + 감으로 우선순위 = 약 3시간/주, 편향 위험
  · After  : 주 1회 자동 수집·분류 → TOP 5 페인포인트 + 해결책 제안 = 읽는 데 5분/주

🔌 연결 소스 (실제 의존만)
  · Firecrawl MCP            → 자사몰·다나와·커뮤니티 웹 리뷰 수집 (실데이터 모드)
  · Gmail 라벨 cs-inquiries  → CS 메일 수집 (실데이터 모드)
  · Google Sheets "VoC 분석" → 5축 분류 결과 적재
  · Notion 인사이트 페이지    → TOP 5 페인포인트 + 해결책 보관
  · Discord webhook          → TOP 3 요약 발송

💡 핵심 포인트
  · 5축 = 감정 (-2~+2) · 카테고리 (제품/디자인/배송/CS/가격) · 긴급도 (0~3) ·
    신규성 (전주 대비) · 임팩트 (매출 영향)
  · LLM 분류의 핵심은 "축과 척도를 명문화"하는 것 — 프롬프트에 스키마가 박혀 있다.
  · 별점과 긴급도는 독립 축 : 별점 5점이어도 펌프 불량이면 긴급도가 높을 수 있다.
  · 🔗 TOP 5 페인포인트는 Part 5 의 landing-copy 가 "고객의 언어"로 페인 섹션을
    쓸 때 그대로 재사용된다 (체이닝).

👉 다음 (2단계 · 실행) 으로 갈까요?
```

📐 **스킬? 에이전트?** → 도구(Gmail·Sheets·Notion) + 트리거(주 1회) + 산출(Discord)이 있으니 🤖 **에이전트**. 만드는 법은 3-1의 **에이전트 5요소** 골격 그대로.

⏸ 답 받고 2단계로.

### 2-1 작동 구조

```
🔄 에이전트 워크플로
  [자사몰 리뷰 + 올리브영 + 인스타 멘션 + X 검색 + Gmail CS]
    → 최대 500건 수집 (실전) / 40건 (샘플)
    → 5축 분류 (감정 · 카테고리 · 긴급도 · 신규성 · 임팩트)
    → 페인포인트 클러스터링 → TOP 5 + 해결책 제안
    → Google Sheets 적재 + Notion 인사이트 + Discord TOP 3

📄 에이전트 정의에서 새로 볼 부분 (agents/part4-research/voc-analyzer.md):
  ① 5축 분류 스키마가 프롬프트에 정의됨 : 축과 척도의 명문화
  ② 다출처 수집 : 출처마다 포맷이 달라도 같은 스키마로 정규화
  ③ 긴급도 3 (환불 지연·오배송) 은 주간 리포트와 별개로 즉시 플래그
```

### 2-2 데이터 준비 (모드 선택)

```
📊 데이터 모드:

  [A] 샘플 데이터 모드 ⭐ 추천 · 촬영 기본
      sample-data/voc-reviews.csv — 결로아 리뷰·멘션·CS 40건 픽스처.
      (자사몰 16 · 올리브영 9 · 인스타 6 · X 5 · CS메일 7 내외 · 5출처 혼합)
      긍정·부정·중립과 5개 카테고리가 고르게 섞여 분류 시연에 최적.
      ⚠ 픽스처는 수정 없이 그대로 로드 — 라이브 500건 대신 40건으로 결정적 재현.

  [B] 실데이터 모드 (Firecrawl 웹 수집)
      제품명/리뷰 페이지 URL + Gmail cs-inquiries 라벨에서 최근 리뷰·문의 수집.
      · firecrawl_search 로 채널 발굴 → firecrawl_scrape 로 본문 수집
      · 수집량에 따라 5~15분 소요 (실전은 500건 규모).
```

> ⚠ **웹 리뷰 수집 현장 노하우** (Firecrawl 실전 검증)
> · **쿠팡** : Akamai 봇 차단 → Firecrawl·헤드리스 전부 실패. 우회 = 실 Chrome 화면 캡처 또는 다나와·네이버 미러.
> · **다나와** : 상품 페이지의 리뷰는 AJAX 지연로딩이라 본문이 안 잡힘 →
>   리뷰 AJAX 엔드포인트를 직접 스크랩 :
>   `prod.danawa.com/info/dpg/ajax/companyProductReview.ajax.php?t=companyReview&prodCode={pcode}&page=1&limit=30&score=0&sortType=BEST`
>   (G마켓·옥션 등 제휴몰 상품평이 별점·날짜·작성자와 함께 텍스트로 추출됨)
> · **자사몰(카페24·아임웹 등)** : 구매평·Q&A는 보통 로그인/AJAX 게이트 → 본문이 비어 나옴. 별도 리뷰 위젯 API 필요.
> · **네이버 블로그** : `firecrawl_search` + `includeDomains:["blog.naver.com"]` 은 노이즈가 큼 → 제품 고유명사로 직접 URL 확인 권장.

[A] 선택 시 : ① 40건 출처 분포 표 → ② Google Sheets **"VoC 분석"** 생성 (원문 + 5축 컬럼)
→ ③ 미리보기 5건 표시.
[B] 선택 시 : 제품명/리뷰 URL·검색어·Gmail 라벨 차례로 질문 → Firecrawl 수집 실행.

### 2-3 실제 실행

```
▶ 에이전트 워크플로 단계별 실행:

  1. 40건 로드 + 정규화
  2. 5축 분류 (하이라이트 : 대표 5건의 분류 근거를 풀어서 설명)
       예 : "환불 5일째 안 들어와요" → 감정 -2 · CS · 긴급도 3 · 임팩트 高
  3. 분류 결과 Google Sheets 적재               → 40행 + 축별 분포 요약
  4. 클러스터링 → TOP 5 페인포인트:
       ① 용기·펌프 불편 (7건) : 펌프 과다·입구 좁음·펌프 불량 → 개선: 패드형/리필 검토
       ② 배송·환불 처리 지연 (5건) : 긴급도 高 → CS 프로세스 점검
       ③ 향 호불호 (3건) → 무향 라인 검토
       ④ 가격 부담 (3건) → 세일 주기·구독 혜택 커뮤니케이션
       ⑤ 챌린지 운영 이슈 (3건) : 쿠폰 미지급·앱 연동 → 이벤트 운영 개선
  5. Notion 주간 인사이트 페이지 생성 (TOP 5 + 해결책)
  6. Discord TOP 3 embed (빈도 + 감정 추이)
```

⏸ 각 단계 결과 확인 후 다음 단계. 실행 다 보면 3단계로.
