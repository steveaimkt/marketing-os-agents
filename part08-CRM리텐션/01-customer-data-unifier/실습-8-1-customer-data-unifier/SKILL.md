# Part 8 / 8-1 · 고객 데이터 통합 에이전트 실습 : 소개 → 실행 → 활용

> 자사몰·CS·SNS·광고에 흩어진 고객을 **이메일/전화 매칭으로 1명으로 통합**해 Customer Master DB 를 만든다.
> 핵심 학습 목표 : ① 정규화(이메일·전화·이름) ② 매칭 우선순위 이메일>전화>이름 ③ 개인정보 안전 원칙.
> ⛔ 자동 실행 금지. 단계마다 사용자 답을 받은 뒤에만 다음으로.

## 진행 흐름 (0 → 3 단계)

```
0단계  문제  — 이 데이터에 심긴 문제 5종 중 무엇을 풀지 먼저 고른다   ⏸ 답 대기 (시작 게이트)
1단계  소개  — 하는 일 + Before/After + 연결 소스 + 핵심 포인트       ⏸ 답 대기
2단계  실행  — 작동 구조 + 데이터 준비 + 실제 실행 (매칭 → 마스터)    ⏸ 답 대기

```

## 2단계 · 실행

### 2-0 · 사전 점검
1단계 표에서 ❌ 를 확인한다. 샘플 모드면 ①②③ ❌ 라도 그대로 진행 가능.
⚠ 실데이터 모드로 갈 경우 : 시트 공유 범위·화면 녹화 시 개인정보 노출을 먼저 점검한다.

### 2-1 · 작동 구조
```
  [4개 소스] 주문 CSV · CS(Gmail 라벨) · SNS 멘션 · 광고 전환
    → 정규화 : 이메일 소문자·공백 / 전화 숫자만 / 이름 공백·한자 제거
    → 매칭 (이메일 > 전화 > 이름) : 같은 사람을 1 customer_id 로
        · 이름만 있는 건 = 저신뢰 후보 → 사람 확인 플래그
        · 어떤 키도 없으면 = 미매칭, 억지 병합 금지
    → 집계 : first/last_order · order_count · total_revenue · cs_count · sns · last_contact
    → 세그먼트 판정 (마지막 주문일 기준) : VIP / Growing / At-Risk / Dormant / 리드
    → Google Sheets "고객 마스터 DB" 적재 + (선택) 1인 고객 360 HTML
```
📄 에이전트 정의에서 새로 볼 부분 (`agents/part8-crm/customer-data-unifier.md`):
  ① 정규화 규칙 — 이메일/전화/이름 처리가 프롬프트에 명문화됨
  ② 매칭 우선순위 — "이메일 > 전화 > 이름" 이 애매한 병합의 판단 기준
  ③ 안전 원칙 — "개인정보 노출 금지 · 리포트 외부 발송 금지" 선언

퀴즈: "이름이 `김 서연`(공백)과 `김서연`, 이메일이 같으면?"
(답: 같은 사람. 이메일이 최우선 키 · 이름은 정규화해 표기 통일)

### 2-2 · 데이터 준비

```
📊 데이터 모드:

  [A] 샘플 데이터 모드 ⭐ 추천 · 촬영 기본
      sample-data/ 4개 소스 (결로아 가상). 원본 44행이 고객 12명 + 미매칭 1건으로 통합.
      의도적으로 심은 중복 3케이스로 매칭 우선순위(이메일>전화>이름)가 결정적으로 재현됨.
      · 사례 A 김서연 : 4개 소스가 같은 이메일 → 1명 (이름 `김 서연` 공백까지 정규화)
      · 사례 B 이지은 : CS·SNS 가 이메일 없이 전화만 → 전화로 병합
      · 사례 C 배아린 : 주문 0인데 CS·광고로 레코드 생성 + SNS 이름-only 저신뢰
      정답지 = sample-data/expected-master.md. ※ 픽스처·정답지는 수정 금지(재현성).

  [B] 실데이터 모드
      본인 주문 CSV + Gmail cs-inquiries 라벨 + (선택) voc-analyzer SNS 결과.
      ⚠ 개인정보 : 시트 공유 권한·화면 노출을 반드시 사전 점검 후 진행.
```

⏸ A/B 답 받고 2-3 으로.

### 2-3 · 실제 실행
```
▶ 에이전트 워크플로 단계별 실행:

  1. 4개 소스 로드 (주문 20 · CS 8 · SNS 6 · 광고 10 = 44행)
  2. 정규화 로그                        → 전화 010-2345-6789 → 01023456789 등 (시연 하이라이트)
  3. 매칭 판정표                        → 44행이 어떤 키로 누구에게 붙는지
       · 김서연 = 이메일 매칭 4소스 → 1명 (C001)
       · 이지은 = 전화 매칭 (이메일 공란 2건 포함) → 1명 (C002)
       · 배아린 = 이메일 매칭 + SNS 이름-only 저신뢰 ⚠ → 검토 플래그
       · @glow_mina = 미매칭 → "검토 대기" (억지 병합 안 함)
  4. Customer Master DB 12행 생성        → customer_id/name/email/phone/first·last_order/
       order_count/total_revenue/ltv_segment/cs_inquiry_count/sns_mentions/last_contact_date
       → Google Sheets "고객 마스터 DB" 적재
  5. (선택) 1인 고객 360 HTML           → outputs/{날짜}/customer-360-{id}.html
       · 타임라인(첫주문→최근→CS→멘션) + 추천 액션 (html-report-template 스킬)
  6. 안전 체크                          → 시트 공유 권한 점검 · 외부 발송 금지 재확인

  🔎 인사이트 : 강도윤(C009)은 Dormant(201일 무주문)지만 SNS 07-01 재접촉 → 8-3 윈백 우선.

  ★ focus case 마무리 : 0단계에서 고른 문제(①~⑤)가 이 실행에서 어떤 키로 어떻게 풀렸는지
     한 줄로 되짚는다. (예 ③을 골랐다면 "이지은 = 이메일 공란 2건을 전화로 병합 → C002")
     ⓪ 전체를 골랐다면 5종을 위 매칭 판정표로 한 번에 확인.
```

⏸ 마스터 DB 12행 확인 후 "3단계" 라고 답해 주세요.
