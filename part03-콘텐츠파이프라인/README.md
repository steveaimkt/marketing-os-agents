# Part 3. 파이프라인 — 콘텐츠 제작·배포를 잇는 자동 흐름

> 이 Part에서 배우는 것: **트렌드 → 콘텐츠 → 발행** 흐름을 끊김 없이 자동화. 첫 번째 진짜 에이전트를 만든다.
> (이 흐름의 다음 단계인 **광고 성과 분석**은 Part 6으로 이어진다 — Part 3은 제작·배포까지.)

## 왜 필요한가

- 1인 마케터가 매주 같은 흐름을 손으로 반복: "트렌드 본다 → 글 쓴다 → 발송한다"
- 흐름이 끊기는 지점이 **결국 시간의 80%**: 트렌드 자료를 캡처해 글로 옮기는 단계, 글을 채널별 톤으로 변형하는 단계
- 에이전트로 묶으면 흐름이 한 번에 흐름

## 이 Part의 산출물 (2 에이전트 + 1 스킬)

| # | 에이전트/스킬 | 입력 | 출력 | 트리거 |
|---|---|---|---|---|
| 1-1 | **email-newsletter** (에이전트) | Notion DB의 이번 주 트렌드 | Gmail 뉴스레터 발송 + 노션 아카이브 | 매주 월 09:00 |
| 1-2 | **content-publisher** (에이전트) | 콘텐츠 캘린더 DB의 Draft 항목 | Buffer 큐에 채널별 변형 게시 | `/publish-this-week` 명령 |
| 1-3 | **newsletter-writing** (스킬) | 자료 5개 + 톤 가이드 | 본문 800자 + 제목 5종 + 요약 1줄 | 다른 에이전트가 호출 |

## 핵심 개념 3가지

### 1. 에이전트 = **시키면 알아서 일하는 워크플로**
- 일반 명령: 한 번 답하고 끝
- 에이전트: 정해진 입력·출력·트리거를 가진 재사용 단위

### 2. 스킬 = **반복되는 글 작성 패턴**
- "뉴스레터는 이렇게 쓴다"는 절차서
- 여러 에이전트가 같은 스킬을 호출 가능 (DRY)

### 3. 파이프라인 = **에이전트들의 체이닝**
- 예: `trend-scanner` → `newsletter-writing` 스킬 → `email-newsletter` → 발송
- 이런 체이닝이 Part 10 AX 팀의 기초

## 사전 준비물 (Part 2까지 완료 상태)

- Gmail MCP (Claude.ai 통합) — `email-newsletter`에 필요
- Buffer MCP — `content-publisher`에 필요
- Notion MCP — 두 에이전트 모두 사용
- Discord Webhook — 발송 후 알림

## 체크리스트 (Part 3 종료 후)

- [ ] `email-newsletter` 에이전트를 호출해 본인 이메일로 뉴스레터 1건 발송 성공
- [ ] `content-publisher`가 Notion 캘린더의 Draft를 Buffer 큐에 옮기는 데 성공
- [ ] `newsletter-writing` 스킬을 다른 명령에서 재호출 가능
- [ ] 디스코드 채널에 발송 알림이 자동 도착
- [ ] 에이전트 정의 파일(`agents/part3-content/*.md`) 구조를 외움
