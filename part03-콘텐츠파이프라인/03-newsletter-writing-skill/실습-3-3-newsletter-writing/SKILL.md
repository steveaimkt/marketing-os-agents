# Part 3 / 3-3 · 뉴스레터 작성 스킬 실습 : 스킬 정의부터 재사용 검증까지

> **여러 에이전트가 공통으로 호출하는 글쓰기 절차서**를 이해하고 직접 가동한다.
> 핵심 학습 목표 : ① 스킬 vs 에이전트 vs 명령의 구분 ② SKILL.md 의 입출력 스키마 ③ DRY 재사용.
> ⚠️ 층위 주의 : 지금 발동된 이 스킬은 "실습 진행용"이고, 학습 대상은 `skills/newsletter-writing` 이다.
> ⛔ 자동 실행 금지. 단계마다 사용자 답을 받은 뒤에만 다음으로.

## 진행 흐름 (3 단계)

```
1단계  소개   스킬 기능 + 연결된 소스 + 개념 구분               ⏸ 인사
2단계  실행   워크플로→스킬 5단계로 SKILL.md 작성 → 호출 → 재사용 검증   ⏸ 답 대기
3단계  활용   내 업무 적용법 + 산출물 + 다음                     ⏸ 선택
```

## 1단계 · 소개

**하는 일** : 자료 몇 건 + 톤 가이드만 넣으면 동일 품질의 뉴스레터(본문 800자 + 제목 5종 + 요약 1줄)를 1분 만에 만드는 **재사용 글쓰기 스킬**을 직접 가동한다.

🔌 **연결된 소스** (실제 의존하는 것만)

```
실습-3-3-newsletter-writing (이 실습 스킬)
├─ 학습 대상 스킬 : skills/newsletter-writing/SKILL.md   ← 직접 호출 대상
├─ 재호출 에이전트 : agents/email-newsletter (3-1)        ← 워크플로 3단계에서 동일 스킬 호출
├─ 입력 픽스처 : sample-data/newsletter-inputs.md        ← 결로아 자료 3건 + 톤 + 페르소나
└─ MCP : 없음 (순수 생성 스킬 · 외부 시스템 불필요)
```

📐 **스킬 vs 에이전트 vs 명령** (Part 3 의 마지막 퍼즐)

```
|        | 에이전트            | 스킬               | 명령             |
|--------|--------------------|--------------------|------------------|
| 목적   | 워크플로 전체       | 작업 패턴          | 단일 호출        |
| 위치   | agents/            | skills/            | commands/        |
| 예     | email-newsletter   | newsletter-writing | /send-newsletter |

→ "뉴스레터는 이렇게 쓴다"는 패턴이므로 스킬. 에이전트는 이 스킬을 도구처럼 호출한다.
```

🔑 **핵심 포인트**
- 반복되는 글쓰기 패턴은 스킬로 한 번 만들고 영원히 재사용 (DRY).
- 스킬은 "어떻게 쓰는가"만 정의 · "언제·무엇으로"는 호출하는 쪽(에이전트)이 정한다.
- ⏱ Before : 톤 흔들림 + 제목 고민 15분 + 본문 40분 → After : 자료만 넣으면 약 1분.

```

```

## 2단계 · 실행

### 2-0. 사전 점검 (조용히)

```bash
test -f "skills/newsletter-writing/SKILL.md" && echo "스킬 정의 ✅"
```

| # | 항목 | 용도 | 상태 |
|---|---|---|---|
| ① | skills/newsletter-writing/SKILL.md | 학습 대상 스킬 정의 | {✅ / ❌} |
| ② | 외부 MCP | 불필요 (순수 생성 스킬) | 해당 없음 |

❌ 인 경우 → 본 실습의 하이라이트로 전환 : 2-1 에서 SKILL.md 를 **빈 파일부터 함께 작성**한다
(frontmatter → 입력 스키마 → 출력 스키마 → 검증 체크리스트 순, 요소마다 ⏸ 확인).

### 2-1. 워크플로 → 스킬 변환 5단계 (이 실습의 본론)

> "매주 손으로 하던 뉴스레터 워크플로"를 SKILL.md 한 장으로 박제하는 과정을 **직접 밟는다**.
> 완성본 `skills/newsletter-writing/SKILL.md` 는 각 단계의 **정답지**로만 비교해서 연다.

```
① 반복 신호 포착 — "이거 또 하네?"
   판별 질문 : "다음 주에도 똑같은 절차로 또 하나?" → Yes = 스킬감
   → 뉴스레터는 매주 = 100% 스킬 후보

② 변하는 것 / 안 변하는 것 가르기 ← 설계의 전부
   ┌ 매주 바뀜 → 입력(parameters) : 자료 5건 · 톤 · 타겟 · CTA
   └ 매주 같음 → 스킬 본문(절차·원칙) : 두괄식 · 숫자 · 2인칭 · CTA 1개 · 금기어 제거
   → "바뀌는 건 입력으로 빼고, 안 바뀌는 건 스킬에 박는다"

③ SKILL.md 4요소를 빈 파일에 위→아래로 작성
   1) frontmatter  : name + description   ← description 이 가장 중요
                     (Claude 가 "언제 이 스킬 쓸지" 고르는 유일한 단서)
   2) 입력 스키마  : ②의 '바뀌는 것'을 yaml 로 (sources·brand_voice·audience·cta)
   3) 출력 스키마  : 본문 800자 + 제목 5종 + 요약 1줄 + CTA 1문장 (받아쓰기 좋게 '모양' 고정)
   4) 검증 체크리스트 : 끝내기 전 자체 점검 (금기어 0 · 분량 · CTA 1개)

④ 머릿속 암묵지를 글로
   "난 원래 이렇게 써"(제목 5패턴·금기어·이모지 정책)를 글쓰기 원칙 섹션에 명문화
   → 누가 호출해도 같은 품질이 나오는 이유

⑤ 호출 테스트 → 에이전트 결합  (→ 2-3 에서 실제로 실행)

  에이전트 정의와의 차이 : 스킬엔 tools·trigger 가 없다. "어떻게 쓰는가"만 정의한다.
```

**시연 방식** : ①②를 사용자와 말로 정리 → ③ 빈 SKILL.md 를 요소별로 함께 채움
(요소 1개 쓸 때마다 ⏸) → 완성본을 `open` 해 **내가 쓴 것 ↔ 정답지** 대조.

```bash
open "skills/newsletter-writing/SKILL.md"   # 정답지 비교용 (작성 후에 연다)
```

퀴즈: "Claude 가 스킬을 자동 선택할 때 보는 필드는?" (정답: description)

### 2-2. 데이터 준비

**[A] 샘플 모드 (기본 · 촬영용)** — `sample-data/newsletter-inputs.md` 를 읽어 입력 4종을 표로 정리:

```
📥 스킬 입력 (4종) · 결로아(Gyeolloa) · 3-1 weekly-trends.json 과 동일 출처
  · sources     : 자료 3건 (올영 토너 매출 +42% / NN일 챌린지 해시태그 2.1배 / 저자극 리뷰 12만건)
  · brand_voice : 친근+신뢰 · 핵심 어휘 6개(케어·챌린지·루틴·가이드·저자극·결) · 금기어 4개(미백·주름 개선 등 화장품법)
  · audience    : 25~34 여성 · 모공·유분 고민 · 모바일 스캔 독자
  · cta         : "21일 모공 챌린지, 이번 주 시작입니다" + https://gyeolloa.kr/challenge-21d
```

**[B] 실데이터 모드** — 자료 URL 3~5건 + 톤 형용사 3개 + 타겟 한 줄 + CTA 를 차례로 질문.

### 2-3. 실제 실행

```
▶ newsletter-writing 스킬을 직접 호출합니다 (2-2 입력 그대로):

  1. 스킬 호출                          → 생성 진행
  2. 출력 4종 표시:
     · 본문 800자 (자료 3건이 어디에 어떻게 엮였는지 하이라이트)
     · 제목 5종 (앵글 라벨 : 긴급성 / 호기심 / 권위 / 사회적 증거 / 혜택)
     · 1줄 요약 (프리헤더 60자 이내)
     · CTA 1문장
  3. 검증 체크리스트 자동 실행          → 금기어 0건 · 분량 통과 여부 표시

▶ 재사용성 검증 (DRY 의 증명) : **같은 스킬을 에이전트가 호출**
   email-newsletter 에이전트(3-1)의 워크플로 3단계가 동일 스킬을 호출한다.
   (3-1 실습을 했다면 그때 본문과 지금 직접 호출 결과가 같은 스킬에서 나왔음을 비교)
```

수정 피드백 루프: "제목을 더 직설적으로" 같은 한 줄 피드백 → 해당 부분만 재생성 (1회 시연).

⏸ 결과 확인하고 답 받은 뒤 3단계로.

## 3단계 · 활용 — 지금, 내 업무로

> 안내로 끝내지 않는다. **당신이 매번 반복하는 글쓰기 작업을 지금 스킬로 박제해본다.**

### 적용 게이트 (능동 질문 ⏸)

```
🚀 그럼, 지금부터 내 업무에 적용해볼까요?

   뉴스레터처럼 "매번 같은 절차로 반복하는 글쓰기"가 당신에게도 있을 거예요.
   아래 3가지만 알려주시면, 지금 바로 그 작업용 SKILL.md 4요소 초안을 같이 만들어 드릴게요.

   ① 매번/매주 반복하는 글쓰기 작업 1개는?   (예: 상품 상세 소개, 블로그 도입부)
   ② 그 작업에서 '매번 바뀌는 입력'은?        (예: 상품명·핵심 혜택·타겟)
   ③ '항상 지키는 규칙·말투'는?               (예: 3문장 요약 후 CTA, 존댓말)
```

→ 답을 받으면 : ②=입력 스키마 / ③=글쓰기 원칙으로 정리 → 그 작업용 **SKILL.md 4요소 초안 즉석 생성**
   → 샘플 입력으로 1회 호출해 작동 확인까지.
→ "건너뛰기" 하면 아래 확장 가이드만 안내하고 마무리.

### 오래 쓰는 법 (적용 후 정착)
1. **다른 스킬로 확장** — 같은 4요소 골격(frontmatter→입력→출력→검증)으로 SNS 캡션·블로그 도입부 스킬을 직접 만든다.
2. **에이전트에서 재호출** — content-publisher 등 다른 에이전트의 워크플로 한 단계에 `Skill(newsletter-writing)` 으로 끼워 DRY 재사용.
3. **톤 커스터마이즈** — `brand_voice` 의 핵심 어휘·금기어만 내 브랜드로 교체하면 동일 스킬이 우리 톤으로 작동.
4. **품질 체이닝** — 생성된 제목 5종을 quality-reviewer 6축 채점에 넘겨 자동 검수 (Part 5 예고).
